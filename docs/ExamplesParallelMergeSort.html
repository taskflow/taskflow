Parallel Merge Sort

We implement parallel merge sort using Taskflow's runtime tasking, demonstrating how tf::Runtime enables efficient divide-and-conquer parallelism where each recursive level fans out work to available workers without blocking any thread.

Problem Formulation

Merge sort divides an array into two halves, recursively sorts each half, then merges the two sorted halves. The two recursive calls are independent of each other and can therefore run in parallel. The merge step must wait for both halves before it can proceed.

The sequential algorithm is:

void merge_sort(std::vector<int>& data, int left, int right) {
if(right - left <= 1) return;
int mid = left + (right - left) / 2;
merge_sort(data, left, mid);
merge_sort(data, mid, right);
std::inplace_merge(data.begin() + left,
data.begin() + mid,
data.begin() + right);
}

Parallel Implementation

We parallelise the recursive sorting step using tf::Runtime::silent_async. When the subrange is small enough (below a cutoff threshold), we fall back to sequential sorting to avoid creating more tasks than the scheduler can efficiently handle. tf::Runtime::corun waits for the spawned subtask cooperatively — the calling worker stays active rather than blocking:

#include <taskflow/taskflow.hpp>
#include <algorithm>
void parallel_merge_sort(
std::vector<int>& data, int left, int right,
tf::Runtime& rt, int cutoff = 1024
) {
if(right - left <= cutoff) {
// base case: fall back to sequential sort for small ranges
std::sort(data.begin() + left, data.begin() + right);
return;
}
int mid = left + (right - left) / 2;
// spawn the left half asynchronously
rt.silent_async([&data, left, mid, cutoff](tf::Runtime& rt1) {
parallel_merge_sort(data, left, mid, rt1, cutoff);
});
// sort the right half inline (tail optimisation — avoids one extra task)
parallel_merge_sort(data, mid, right, rt, cutoff);
// cooperatively wait for the left half to finish
rt.corun();
// merge the two sorted halves
std::inplace_merge(
data.begin() + left,
data.begin() + mid,
data.begin() + right
);
}
int main() {
const int N = 1 << 20; // 1M elements
std::vector<int> data(N);
std::generate(data.begin(), data.end(), std::rand);
tf::Executor executor;
executor.silent_async([&data, N](tf::Runtime& rt) {
parallel_merge_sort(data, 0, N, rt);
});
executor.wait_for_all();
assert(std::is_sorted(data.begin(), data.end()));
std::cout << "sorted " << N << " elements\n";
return 0;
}

Key Design Points

Cutoff threshold.** Recursively spawning tasks all the way to single-element ranges would create millions of tasks with negligible work each. The cutoff threshold switches to std::sort for small subranges, keeping the number of tasks proportional to the available parallelism rather than the input size. A good rule of thumb is to set the cutoff so that the sequential base case takes at least a few microseconds — typically a few thousand elements for integer sorting.

Tail optimisation.** Only the left half is spawned as an async task; the right half is computed inline in the current execution context. This halves the number of tasks at every level of the recursion tree and avoids the overhead of creating and scheduling a task that could run directly.

Cooperative waiting.** tf::Runtime::corun does not block the calling worker — it participates in the work-stealing loop while waiting for the left-half task to complete. This is essential for recursive parallelism: if the worker blocked, it would hold a thread hostage while the left-half task waits for a free worker, potentially causing deadlock or severe under-utilisation.

Benchmarking

Runtime comparison for sorting 1M random integers on a 12-core machine:

Algorithm Time
std::sort (seq) 180 ms
Parallel merge sort 38 ms
Speed-up ~4.7×

The speed-up is sub-linear at this size because std::inplace_merge at the top levels of the tree is sequential and dominates the runtime as the recursive parallelism collapses. For larger inputs or with a parallel merge step, the speed-up approaches the number of cores.
